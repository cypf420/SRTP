# 工程计划（中文版）
## 面向工具调用错误因果可解释性研究的工程化实施方案

这份计划把你的研究目标拆成了**可执行、可验收、可排期**的工程动作。目标不是“看起来完整”，而是让你在一个学期内真正跑出：

- 一套实验框架
- 一份可用数据集
- 一条可复现因果分析链路
- 一套表征边界分析链路
- 一个轻量防御系统原型
- 一篇可投稿论文

---

## 一、总目标拆解

你的总目标可以分成两个并行结果：

### 结果 A：论文成果
你要证明：
1. 工具调用错误不是随机现象；
2. 它们受一组可解释特征驱动；
3. 因果图可以找出关键驱动因子；
4. 表征边界可以解释哪些样本最容易从“正确调用”滑向“错误调用”；
5. 基于这些因子可以构建防御系统。

### 结果 B：系统成果
你要做出一个可接入任意 tool-calling LLM 的前置防御层，能够：
- 检测高风险请求；
- 对参数缺失做澄清；
- 对容易混淆的工具描述做增强；
- 标记边界附近的脆弱样本；
- 降低工具选错和参数幻构。

## 三、逐阶段工程任务拆解

# Phase 0：项目初始化与可复现实验底座（1~2天）

## 目标
建立一个后续不会反复推倒重来的工程骨架，并提前把“可复现”和“可追踪”能力铺好。

## 具体动作

### 动作 0.1：初始化仓库
- 创建项目根目录并初始化 Git。
- 新建 `README.md`，写清研究目标、目录说明、运行入口、数据说明和当前研究边界。
- 新建 `.gitignore`，忽略 `.env`、`__pycache__/`、`data/raw_runs/`、模型缓存、临时图像和日志文件。
- 新建 `requirements.txt` 与 `requirements-dev.txt`，区分运行依赖和开发依赖。
- 新建 `.env.example`，列出 `OPENAI_API_KEY`、`OPEN_MODEL_ENDPOINT`、`OPEN_MODEL_NAME`、`HF_HOME` 等变量。
- 约定命名规范，例如 `snake_case` 文件名、`run_YYYYMMDD_xxx` 形式的实验目录名。

### 动作 0.2：建立基础目录
建议目录结构：

```bash
project/
├── configs/            # 实验配置
├── tools/              # 工具 schema
├── tasks/              # 用户任务模板
├── prompts/            # Prompt 模板
├── runner/             # 模型调用与批处理
├── evaluator/          # 自动评估器
├── features/           # 特征提取
├── causal/             # 因果分析代码
├── boundary/           # 表征边界分析
├── defense/            # 防御模块
├── data/
│   ├── raw_runs/
│   ├── processed/
│   ├── annotations/
│   └── external_eval/
├── notebooks/
├── scripts/
└── paper/
```

- 先把空目录建出来，不要等到后期再临时补。
- 在 `configs/`、`tools/`、`tasks/`、`data/processed/` 下放一个占位文件，避免空目录丢失。
- 在 `paper/` 目录放一个 `outline.md`，先固定论文结构骨架。

### 动作 0.3：安装依赖
建议第一版依赖：
- `openai`
- `transformers`
- `torch`
- `pandas`
- `numpy`
- `scikit-learn`
- `sentence-transformers`
- `causal-learn`
- `gcastle`（可选）
- `networkx`
- `matplotlib`
- `seaborn`
- `umap-learn`
- `scipy`
- `statsmodels`
- `pydantic`
- `pyyaml`
- `jsonschema`
- `tqdm`

- 先安装最小运行版本，不要一上来追最新依赖。
- 用一个 `scripts/check_env.py` 脚本逐个 import，确认环境没有缺包。
- 把 `pip freeze` 导出到一个锁定文件，后面复现实验时直接参考。

### 动作 0.4：准备模型接口
- 配置 GPT-4o API。
- 配置一个开源模型调用方式（本地或 vLLM）。
- 这个开源模型必须支持导出 hidden states 或 logits，因为后续表征边界分析会用到。
- 写统一接口，例如 `generate_tool_call(messages, tools, config)`。
- 在接口里统一处理超时、重试、异常日志、返回结构标准化。
- 写一个最小测试脚本，保证可以输入 tools + user query，拿到结构化 tool call 输出。
- 对开源模型再补一个最小测试，确认可以拿到中间表示或 logits。

### 动作 0.5：建立实验配置与追踪规范
- 在 `configs/models/` 下给每个模型写一个 YAML 配置。
- 在 `configs/runs/` 下写一个默认运行配置，包含 `seed`、`temperature`、`top_p`、`max_tokens`、`batch_size`。
- 明确每次实验都要记录：模型版本、prompt 版本、工具版本、任务版本、时间戳、Git commit hash。
- 规定输出目录结构，例如 `outputs/run_20260407_gpt4o_v1/`。
- 先手工跑一次小实验，确认日志字段足够后续排错和复现。

## 验收标准
- 项目可以 `git clone` 后一键安装依赖。
- 至少一个闭源模型和一个开源模型能完成最小 tool call 测试。
- 开源模型至少能在小批量样本上导出 hidden states 或 logits。
- 每次实验都能自动生成独立的 `run_id` 和配置快照。

# Phase 1：文献摸底与问题定型（第1~2周）

## 目标
明确你的工作与现有工作之间的差异，形成论文的“问题定位”，同时收敛你的可证明主张与不该过度主张的边界。

## 具体动作

### 动作 1.1：建立文献表
新建一个 CSV/Excel 表格，字段如下：
- `paper_title`
- `year`
- `venue`
- `task_type`
- `problem_definition`
- `dataset`
- `method`
- `metrics`
- `是否研究工具选择错误`
- `是否研究参数幻构`
- `是否研究澄清行为`
- `是否有因果解释`
- `是否分析内部表征`
- `局限性`
- `你可借鉴的部分`

- 每读完一篇，至少写 3 句摘要，不要只存标题。
- 对每篇文献标记“可作为 baseline / 可作为 related work / 仅作背景”的角色。

### 动作 1.2：关键词搜索
分别搜索以下方向，每个方向收集 5 篇左右：
- tool use failure analysis
- function calling hallucination
- prompt injection against tool use
- agent tool selection benchmark
- causal interpretability in LLMs
- representation analysis / decision boundary in LLMs

- 优先记录近 3 年文献。
- 把公开代码、公开数据、是否可复现也一起记下来。
- 对明显高相关的论文，额外下载 PDF 并做精读标记。

### 动作 1.3：写逐篇阅读笔记
- 对高相关论文单独写一页笔记。
- 每页至少回答四个问题：
1. 它研究的具体对象是什么？
2. 它的数据和标签是什么？
3. 它的方法为什么有效？
4. 它离你的题目还差什么？

- 读到可以直接复用的指标、数据格式、图表形式，就单独摘出来。
- 如果某篇论文与你的工作最相近，单独写“差异点清单”。

### 动作 1.4：写“研究空白声明”
写一页文档，回答三个问题：
1. 现有工作主要解决了什么？
2. 它们没有解决什么？
3. 你的工作补的是哪块空白？

建议最后收敛成一句话：
> 现有工具调用研究大多停留在性能评测与案例分析层面，缺少对工具选择错误与参数幻构的因果机制解释，也较少分析错误样本在内部表示空间中的脆弱边界；本研究试图建立一套面向工具调用失败的因果可解释性与边界分析框架，并将其转化为可部署的防御层。

### 动作 1.5：收敛论文主张与边界
- 明确你真正要 claim 的内容，例如“在当前数据构造和特征假设下，存在稳定的关键驱动因子”。
- 明确你不 claim 的内容，例如“这不是对所有 agent 场景的最终因果真理”。
- 写一份 0.5~1 页的“研究边界说明”，后面论文引言和局限性部分直接可复用。
- 提前选 1~2 个目标投稿方向，反推实验深度和写作风格。

## 验收标准
- 至少完成 20 篇论文的文献表。
- 至少完成 5 篇高相关论文的详细笔记。
- 写出 1 页研究问题声明。
- 写出 1 页研究边界说明。
- 能用 3 分钟口头讲清楚“为什么这个题值得做”和“你不打算过度声称什么”。

# Phase 2：最小实验框架搭建（第3周）

## 目标
搭一个能稳定跑数据的最小系统，而不是一次性做复杂平台；同时让后续数据、特征、因果、边界分析都能接上。

## 具体动作

### 动作 2.1：定义工具 schema
先草拟 15 个工具候选，再收敛到 10 个工具，保存为 JSON：
- 3 个相似工具
- 4 个参数复杂工具
- 3 个低歧义工具

每个工具包含：
- `name`
- `description`
- `parameters`
- `required`
- `examples`
- `risk_level`（可选）
- `version`

- 先列出所有候选工具及其研究作用，例如“制造混淆”“研究参数幻构”“作为低歧义对照”。
- 对每个工具补一行“容易与谁混淆”的说明。
- 对复杂参数工具补一行“最容易缺失的关键字段”。
- 对高风险工具单独标记，后续用于防御实验。
- 完成后手工检查一次，避免工具之间的差异小到连人都分不清。

### 动作 2.2：定义任务模板
先做 50 个 base task，再自动生成变体。每个任务模板包含：
- `task_id`
- `intent`
- `domain`
- `language`
- `dialog_context`
- `gold_tool`
- `gold_params`
- `need_clarification`
- `distractor_tools`
- `variants`

- 先按工具组分配任务，不要让某几个工具样本过多。
- 每个 base task 先写一条“最标准表达”。
- 再手工给每个 task 增加至少一种模糊说法和一种参数不完整说法。
- 对每个 task 明确：如果缺关键参数，正确行为到底是“澄清”还是“拒绝调用”。
- 先人工检查 20 个任务，看 gold tool 和 gold params 是否真的唯一明确。

### 动作 2.3：冻结 prompt 模板
- 写一个系统 prompt 模板，固定模型的 tool-calling 行为要求。
- 写一个工具注入模板，统一 schema 的插入方式。
- 写一个输出解析约束，确保结构化返回。
- 先只保留 1 个主 prompt 版本，不要一开始就维护太多 prompt。
- 给这个 prompt 命名为 `prompt_v1`，后续修改必须升版本。

### 动作 2.4：写批量 runner
runner 要能完成：
- 读取任务
- 注入工具列表
- 调用模型
- 保存模型输出 JSON
- 记录日志

建议每条记录保存：
- `sample_id`
- `model_name`
- `prompt_version`
- `tool_version`
- `task_version`
- `prompt`
- `tools`
- `raw_response`
- `tool_name_pred`
- `arguments_pred`
- `latency_ms`
- `usage`
- `seed`
- `timestamp`

- 支持按配置文件批量运行，而不是把参数写死在脚本里。
- 支持断点续跑，避免长任务中断后全重跑。
- 对开源模型增加一个可选开关，保存 hidden states 或 logits 到单独目录。
- 如果模型返回非法 JSON，要把原始文本完整保存，后续便于排查。

### 动作 2.5：写自动评估器
规则版 evaluator 判断：
- 工具是否正确
- 参数键是否缺失
- 参数键是否多余
- 参数值是否格式错误
- 是否出现幻构值
- 是否本应澄清却直接调用

输出统一标签，建议分成两层：

第一层主标签：
- `correct`
- `wrong_tool`
- `param_error`
- `should_clarify_but_called`

第二层细标签：
- `missing_required_param`
- `extra_param`
- `param_type_error`
- `param_value_hallucination`
- `high_risk_tool_misuse`

- 先写规则判定，再给“规则拿不准”的样本打上 `needs_review`。
- 对每种标签至少构造 5 条单元测试样本，验证评估器没有明显漏洞。
- 把评估器输出设计成结构化表格，后续直接可接特征和统计分析。

### 动作 2.6：做第一次小规模联调
- 先跑 100 条样本，不要直接冲大规模。
- 手工抽查至少 30 条正确样本和 30 条错误样本。
- 对抽查结果记录三种问题：
1. 工具设计不合理；
2. 任务 gold 标注不清；
3. evaluator 判定规则有误。

- 根据问题修一次工具库、任务模板和评估器。
- 修完后再跑一轮小样本，确认系统趋于稳定。

## 验收标准
- 能一键跑完 100 条样本。
- 能自动生成评估结果 CSV。
- 评估脚本和人工检查结果基本一致。
- 开源模型子集可以额外导出表示，用于后续边界分析。
- `prompt_v1`、`tool_v1`、`task_v1` 被明确冻结。

# Phase 3：数据集构建、标注与外部验证集准备（第4~6周）

## 目标
形成第一版可用于分析的工具调用错误数据集，并提前准备一个不参与调参的外部验证集。

## 具体动作

### 动作 3.1：扩展任务模板
从 50 个 base task 扩展到 200+ 任务意图。

每个任务生成四类变体：
- 清晰直接型
- 模糊歧义型
- 参数不完整型
- 误导干扰型

- 先为每个工具分配最少意图数，避免类别严重不平衡。
- 为每类变体写生成规则，不要完全依赖临时手工改写。
- 生成后对每 20 条自动变体随机抽查一次，防止模板漂移。
- 对中文和英文分别保留一定数量，不要等到最后再补多语言。

### 动作 3.2：确定数据划分策略
- 按 `intent` 或 `template_family` 划分 train/dev/test，而不是按单条 paraphrase 随机划分。
- 确保同一个 base task 的轻微改写不要同时出现在训练和测试中。
- 单独留出一个 `external_eval` 子集，不参与 prompt 调参和 feature 设计。
- 把划分结果固化成 `split.csv`，后续所有实验都复用。

### 动作 3.3：批量跑模型
至少跑两个模型：
- GPT-4o
- 开源模型

目标规模：
- 至少 2000 条样本
- 错误样本不低于 40%
- 至少覆盖两种语言

- 先跑一轮默认配置，拿到基础错误分布。
- 如果错误率过低，再定向增加高混淆任务，而不是盲目堆数据。
- 对开源模型额外保存一部分样本的 hidden states 或 logits，供边界分析使用。
- 明确哪些实验是“正式主结果”，哪些只是“调试跑数”，避免后续结果混淆。

### 动作 3.4：写标注规范并人工复核
- 写 `annotation_guideline.md`，明确每种错误标签的判定条件和边界案例。
- 随机抽取 100~150 条人工检查：
1. 工具标签是否正确；
2. 参数标签是否正确；
3. 是否本应澄清；
4. 评估脚本是否过严或过松。

- 如果只有你一个标注者，至少做一次“延时复标”，间隔几天后对其中 30~50 条重标。
- 把所有争议样本单独存成一个 `adjudication.csv`，记录最终判定理由。

### 动作 3.5：整理 processed 数据
输出至少六份文件：

#### 文件 1：`episodes.jsonl`
存每条原始样本

#### 文件 2：`labels.csv`
存每条样本主标签和细标签

#### 文件 3：`task_metadata.csv`
存任务、工具、语言、变体类型等元信息

#### 文件 4：`run_metadata.csv`
存模型版本、prompt 版本、seed、时间戳等运行信息

#### 文件 5：`split.csv`
存 train/dev/test/external_eval 划分

#### 文件 6：`boundary_subset.jsonl`
存用于表征边界分析的表示可用样本子集

- 所有文件都用同一个 `sample_id` 关联。
- 在 `processed/README.md` 里解释每个文件的字段含义。

### 动作 3.6：写数据卡与局限性说明
- 写 `data_card.md`，说明数据来源、构造方式、标注流程、语言覆盖和限制。
- 明确区分“合成任务”与“真实风格任务”。
- 明确哪些样本来自模板生成，哪些来自手工构造。
- 提前写上偏差风险，后面论文可以直接复用。

### 动作 3.7：准备外部验证集
- 从公开 benchmark、公开示例、真实风格任务表述中整理 100~200 条外部样本。
- 这部分样本不参与 prompt 调整、特征筛选和规则打磨。
- 为外部验证集也建立 gold tool、gold params 和标签规则。
- 在主实验之外单独报告“域内结果”和“外部验证结果”。

## 验收标准
- 数据条数达到预期。
- 数据格式统一。
- 有清晰 train/dev/test/external_eval 划分。
- 有标注规范与数据卡。
- 有一个不参与调参的外部验证集。

# Phase 4：特征工程、因果发现与表征边界分析（第7~10周）

## 目标
把“错误案例”变成“可分析的因果变量系统”，同时建立一条可以解释脆弱样本的表征边界分析链路。

## 具体动作

### 动作 4.1：明确因果假设与变量边界
- 先写一页 `causal_assumptions.md`，列出所有可观测变量和不可观测变量。
- 按时间顺序约束变量：用户请求先于工具定义，工具定义先于模型输出，模型输出先于错误标签。
- 明确哪些边是物理上不可能出现的，后续跑图时作为先验约束。
- 明确哪些变量可能是混杂因素，例如语言、任务难度、工具组别。
- 提前写清楚：这里做的是“在当前数据构造和假设下的因果结构学习”，不是绝对真理。

### 动作 4.2：定义特征表
先列出 20~30 个特征，形成 `feature_schema.yaml`，字段包含：
- `feature_name`
- `group`
- `description`
- `type`
- `extraction_method`
- `value_range`
- `missing_strategy`
- `reliability_note`

建议至少覆盖四组特征：
- 指令侧特征
- 工具侧特征
- 匹配侧特征
- 上下文侧特征

- 每个特征都写清是规则提取、embedding 计算还是 LLM 辅助判别。
- 每个特征都写清“为什么它可能是因子”，不要只列名字。
- 对高成本特征额外标记，避免后续批量运行过慢。

### 动作 4.3：实现特征提取器
分别写四个模块：
- `instruction_features.py`
- `tool_features.py`
- `matching_features.py`
- `context_features.py`

每个模块负责产出固定字段。

- 对每个模块先做 5~10 条样本的单元测试。
- 提取结果要尽量结构化，不要把复杂判断塞成难以解释的大字符串。
- 对 LLM 辅助特征单独做缓存，避免重复计费和重复波动。
- 把每个特征的提取失败情况记录下来，避免静默缺失。

### 动作 4.4：做特征质量检查
- 统计每个特征的缺失率、分布范围和异常值。
- 检查明显的标签泄漏，例如某特征是不是直接编码了 gold 信息。
- 检查多重共线性，避免一堆几乎等价的特征一起进图。
- 随机抽查 50 条样本，对照原文看特征是否真的提对了。
- 如果某个特征稳定性太差，就及时删掉，不要勉强保留。

### 动作 4.5：生成特征矩阵
输出 `features.csv`：
- 每行一个 sample
- 每列一个 feature
- 最后一列是 outcome label

同时补充：
- `feature_stats.json`
- `feature_mapping.json`

- 把类别型变量统一编码。
- 明确数值归一化和离散化规则。
- 记录哪份特征矩阵用于哪组实验，避免版本混乱。

### 动作 4.6：跑基线因果算法
先跑：
- PC
- NOTEARS
- DirectLiNGAM（可选）

输出：
- DAG 图
- 邻接矩阵
- 到错误标签的父节点
- 关键因果路径

- 至少分别对 `wrong_tool` 和 `param_value_hallucination` 做一次分析，不要只看总错误标签。
- 先在小特征子集上跑，确认图不会完全失控。
- 再逐步扩到完整特征集。

### 动作 4.7：做图稳定性与显著性分析
- 对同一组数据做 bootstrap 重采样，多次重复跑图。
- 统计每条边出现的频率。
- 统计哪些父节点在不同 seed、不同模型、不同语言上保持稳定。
- 对关键干预实验使用显著性检验，而不是只看均值下降。
- 形成一张“稳定边清单”，只拿稳定边做核心结论。

### 动作 4.8：做语义对齐增强
加入 embedding 特征：
- 用户请求 embedding
- 工具描述 embedding
- 请求-工具匹配 embedding

通过对齐层融合到原始特征中。

- 先保留一个纯表格特征基线。
- 再做“表格特征 + embedding 对齐”的增强版。
- 对比两者的图结构、稳定性和干预可解释性是否提升。
- 如果 embedding 引入后图更乱、解释更差，就不要强行保留。

### 动作 4.9：做反事实实验
挑 3~5 个被识别为关键因子的特征，人工干预：
- 补足缺失参数
- 重写模糊动词
- 改写工具描述
- 删除误导性上下文
- 降低高混淆工具的相似度

观察错误率是否下降。

- 对每个因子都先定义“如何只改这一点，尽量不改其他条件”。
- 尽量使用配对样本比较，而不是拿两堆不相干样本硬比。
- 把每次干预前后的例子保存下来，后面论文图例会直接用到。

### 动作 4.10：做表征边界分析
- 选择一个能导出 hidden states 或 logits 的开源模型作为边界分析对象。
- 从 `boundary_subset.jsonl` 中取一批正确样本和错误样本。
- 导出指定层表示或 tool-call 前的 logits 表示。
- 对表示做 PCA / UMAP 降维。
- 观察正确调用与错误调用是否形成可分区域。
- 计算边界附近样本的密度、类间距离和错误样本到边界的平均距离。
- 挑出最靠近边界的样本，回看它们对应的指令、工具和特征。
- 检查这些边界样本是否与因果图中的关键因子一致。

### 动作 4.11：做消融实验与鲁棒性测试
至少做以下消融：
- 去掉特征对齐模块
- 去掉人工特征，仅用 embedding
- 去掉工具侧特征
- 去掉匹配侧特征
- 去掉上下文侧特征
- 不做边界分析，只保留因果链
- 不使用边界样本分析，只随机取样

至少做以下鲁棒性：
- 多模型复现
- 中文与英文对比
- 域内测试与外部验证集对比

## 验收标准
- 至少得到一张稳定 DAG。
- 至少找出 3 个可解释关键因子。
- 至少有一个反事实实验显著降低错误率。
- 至少完成一张边界图和一组边界样本分析。
- 至少完成一张消融或鲁棒性对比表。

# Phase 5：防御系统原型（第11~13周）

## 目标
把“因果解释”和“边界脆弱性分析”变成“实际防御能力”。

## 具体动作

### 动作 5.1：做前置澄清模块
逻辑：
- 检测是否缺失关键参数
- 若缺失，则要求澄清
- 否则再调用模型

- 先基于 schema 写一个关键参数检测器。
- 再写一个简单的槽位提取器，判断用户话语里是否真的提供了关键值。
- 为不同工具准备澄清问句模板。
- 设计一个统一输出，例如 `ask_user`, `pass_through`, `block_call`。
- 记录每次触发澄清的原因，后续便于分析误伤。

### 动作 5.2：做工具描述增强模块
逻辑：
- 对语义相似工具计算混淆分数
- 对高混淆工具自动重写 description
- 加示例、加边界说明、加负例提示

- 先从实验结果里提取“最常被混淆的工具对”。
- 给每对工具写出“应该用 / 不应该用”的边界说明。
- 保留原始 description 版本和增强版本，方便做 A/B 对比。
- 对高风险工具额外加权限或适用边界提示。

### 动作 5.3：做边界感知风险评分器
- 结合因果关键因子、工具混淆分数和边界距离，输出一个风险分数。
- 给风险分数分桶，例如 `low / medium / high`。
- 为不同风险等级设定不同行为：
1. 低风险：直接放行；
2. 中风险：增强工具描述；
3. 高风险：先澄清或拦截。

- 先从简单规则开始，不必一开始做复杂学习器。
- 重点是把“解释结果”转成“可执行策略”。

### 动作 5.4：做防御包装器
写成统一接口：

```python
safe_tools, safe_messages, defense_meta = defense_wrapper(tools, messages)
```

- 在包装器里串起：风险检测、澄清模块、工具增强模块、日志记录。
- 包装器不要改动底层模型接口，保持可插拔。
- 输出额外的 `defense_meta`，记录触发了哪些防御动作。
- 让这个包装器既能接 GPT-4o，也能接开源模型。

### 动作 5.5：做系统评估
对比：
- 原始系统
- 加澄清模块
- 加工具增强模块
- 加风险评分器
- 三者都加

评估指标至少包括：
- 工具错误率变化
- 参数幻构率变化
- 总体任务成功率变化
- 正常样本误伤率
- 平均额外澄清轮数
- 推理延迟变化

- 域内测试和外部验证集都要报告。
- 如果某种防御有效但误伤很高，不要只报降错不报代价。

### 动作 5.6：做残余失败分析
- 把加了防御后仍然失败的样本单独拉出来。
- 归纳剩余失败类型，例如：
1. 工具本身定义仍过于接近；
2. 用户请求天然无法判定；
3. 开源模型能力下限太低；
4. 风险评分阈值设置不合理。

- 写一页残余失败总结，后面论文讨论部分直接复用。

## 验收标准
- 工具错误率下降。
- 参数幻构率下降。
- 正常样本误伤率可控。
- 外部验证集上也能看到一定收益。
- 能清楚解释每个防御模块为什么有效、在哪些情况下失效。

## 六、潜在风险与应对策略

### 风险 1：错误样本太少
**应对：** 增加模糊任务和复杂参数任务，主动制造高混淆场景，但要单独保留正常分布的外部验证集，避免全靠“刻意造错”。

### 风险 2：评估器不稳定
**应对：** 先规则，后 LLM 辅助；抽样人工校验；把争议样本单独归档。

### 风险 3：因果图不稳定
**应对：** 减少特征数；先跑小图；做 bootstrap 稳定性统计；只对稳定边下核心结论。

### 风险 4：拿不到闭源模型内部表示
**应对：** 把表征边界分析主要建立在开源模型上；闭源模型继续用于外部行为层评估。

### 风险 5：合成数据偏差过大
**应对：** 写清构造规则；保留真实风格外部验证集；域内结果和外部结果分开报告。

### 风险 6：开源模型 tool call 能力太差
**应对：** 不强求性能一致，把它作为跨模型对比对象；必要时缩小边界分析任务范围。

### 风险 7：防御降错但误伤太高
**应对：** 把澄清模块设置为条件触发，而不是全量拦截；报告误伤率和额外轮数。

### 风险 8：论文创新不够突出
**应对：** 强调“因果解释 + 边界分析 + 数据集 + 防御原型”四位一体，而不是只做单点分析。
