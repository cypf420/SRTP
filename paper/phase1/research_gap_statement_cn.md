# 研究空白声明

## 一、现有工作主要解决了什么

围绕 LLM tool use / function calling，现有研究大致可以分成四类。

第一类是**通用工具使用 benchmark**。这类工作主要回答“模型会不会用工具、能不能在给定工具集合上完成任务”，典型代表包括 API-Bank、ToolLLM、MetaTool、ToolTalk、StableToolBench、CToolEval、ToolSandbox、ToolHaystack 等。它们推动了工具学习从单 API 调用走向多工具、多轮、长上下文和多语言设置，建立了评测基线，也暴露出工具选择、参数补全、长程记忆和执行稳定性等问题。

第二类是**函数调用 / 工具调用能力增强**。这类工作更关注如何训练模型提升 function calling 能力，例如 Granite-Function Calling Model、FnCTOD，以及围绕高质量 synthetic data 或多任务训练的工作。它们证明了工具能力可以通过数据构造、任务分解和多任务训练获得显著提升。

第三类是**安全与失败分析**。这类工作主要研究 prompt injection、间接注入、工具不可用、信息不足和长程脆弱性，例如 InjecAgent、AgentDojo、FAIL-TaLMs、On the Robustness of Agentic Function Calling。它们说明 tool-augmented LLM 的失败并不只是“答错”，而是会在动作层面出现误调用、误执行、错误参数和被恶意上下文劫持。

第四类是**因果可解释性与表示分析**。这类工作不直接研究工具调用，但为本课题提供了方法论基础。例如 Causal Abstraction、Distributed Alignment Search、Causality is Key for Interpretability Claims to Generalise、Do Llamas Work in English 等工作，分别从因果抽象、表示对齐、可解释性主张边界和内部表示空间分析的角度，说明“相关性描述”与“可泛化解释”并不等价。

## 二、这些工作没有解决什么

虽然上述工作已经覆盖了 benchmark、训练、攻击和解释四个方向，但它们之间仍然存在明显断层。

第一，**大多数工具 benchmark 只回答性能问题，不回答机制问题**。它们通常报告成功率、调用正确率、AST 准确率或任务完成率，却很少进一步回答：到底是哪些输入特征、工具定义特征、上下文特征在驱动工具选错或参数幻构。

第二，**失败分析工作通常停留在攻击面或案例面**。像 prompt injection、under-specified query、missing tool 这类工作已经说明模型容易失败，但大多没有把失败系统化为一组“可观测变量 -> 错误类型”的因果分析问题，也没有形成人工可读的特征体系。

第三，**函数调用研究对“应不应该调用 / 是否应先澄清”的关注不够**。很多工作默认输入任务是可执行的，重点放在“选哪个工具”和“参数对不对”，但对于关键信息缺失时模型是否应停止执行、先发起澄清，关注还不充分。

第四，**现有解释工作很少直接落到 tool-calling failure 上**。因果抽象和表示分析为“怎么做解释”提供了理论和方法，但还没有被系统地迁移到“工具选择错误”和“参数幻构”这类具体 agent failure。

第五，**解释结果和防御系统之间缺少闭环**。已有 benchmark 经常能指出模型差，但很少把“关键失效因子”直接转成一个前置防御层，例如条件触发的澄清器、混淆工具描述增强器或边界感知风险评分器。

## 三、你的工作补的是哪块空白

本研究要补的不是“再做一个 benchmark”，也不是“再报一次准确率”，而是补上下面这条目前仍然断裂的链路：

> **从工具调用错误数据，到人工可读特征，到因果结构学习，到表征边界分析，再到可部署的防御模块。**

更具体地说，你的工作补的是四个空白：

1. **面向工具调用错误的结构化失败数据视角。**
   不是只收集成功/失败，而是把每次 tool-call episode 明确拆成请求、上下文、工具集合、gold tool、gold params、错误标签和可解释特征。

2. **面向 wrong tool / parameter hallucination 的因果问题设定。**
   不是只说“这些因素相关”，而是明确研究在当前数据构造和假设下，哪些特征是错误节点的稳定直接父节点，哪些只是伴生现象。

3. **把行为层错误和表示层脆弱性连起来。**
   在因果图之外，再问错误样本是否集中在内部表示空间的边界附近，从而解释为什么某些模糊请求、复杂 schema 或高混淆工具会让样本从正确区滑向错误区。

4. **把解释结果转成防御原型。**
   最终输出不是一张好看的图，而是一个可插拔的 pre-call defense wrapper，用关键因子和边界风险去决定是否放行、澄清或增强工具描述。

## 四、收敛成一句话的研究定位

现有工具调用研究大多停留在性能评测、训练增强或攻击案例分析层面，缺少对工具选择错误与参数幻构的因果机制解释，也较少分析错误样本在内部表示空间中的脆弱边界；本研究试图建立一套面向工具调用失败的因果可解释性与边界分析框架，并将其转化为可部署的防御层。
