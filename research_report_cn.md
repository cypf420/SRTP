# 研究报告（中文版）
## 面向工具调用错误的因果可解释性研究路线图

## 一、研究背景与问题定义

随着大模型从“只生成文本”走向“能够调用外部工具”，安全问题的重心正在发生变化。过去，模型主要是在文本层面出错，例如输出虚假内容、有害内容或无关内容；而现在，模型还可能在**动作层面**出错，例如：

- 本来应该调用 `query_order`，却错误地调用了 `search_web`；
- 本来应该填写订单 ID，却幻构出一个并不存在的 ID；
- 本来应该先澄清缺失参数，却直接执行错误调用；
- 本来应该调用低权限工具，却因为描述混淆调用了高权限工具。

这类错误不只是“回答错了”，而是“做错了事”。因此，单纯的输出安全研究已经不够，我们需要进入**工具调用决策层**，研究模型在工具选择和参数生成上的失误机制。

本课题的核心研究问题可以被精确定义为：

> **在因果意义上，究竟是输入提示词的哪些特征、工具定义的哪些结构、上下文的哪些状态，真正驱动了模型出现“工具选择错误”和“参数幻构”这两类失败行为？**

围绕这一问题，可以再拆成两个子问题：

### 1. 因果驱动问题
当模型调用了错误的工具或生成了错误参数时，真正的驱动因素到底是什么？

可能的来源包括：
- 用户指令表达模糊；
- 工具描述语义过于接近；
- 参数 schema 复杂；
- 工具名称与用户意图对齐不足；
- 中间上下文带来误导；
- 注入式干扰或伪相关信息造成偏移。

你要做的，不是停留在“这些因素经常同时出现”，而是识别**哪些因素是因果上直接推动错误的关键节点**。

### 2. 表征边界问题
除了因果图之外，还需要问另一个问题：

> 模型在内部表示空间中，是否存在“正确调用 / 错误调用”的决策边界？

如果存在，那么：
- 哪些样本最接近这条边界？
- 多语言、模糊请求、复杂工具描述，是否会把本应正确的样本推向错误区域？
- 这些边界附近样本，是否正是系统最脆弱的点？

因此，本研究将采用**因果结构学习 + 表征边界分析**的双路径来解释工具调用失败。

## 二、数据设计

要做好因果解释，必须先有一套足够细、足够结构化的数据。你的数据不能只是“用户问题 + 模型回答”，而必须升级成**轨迹级工具调用数据**。

### 1. 基本数据单元
每个样本至少应包含以下字段：

- 用户原始请求
- 对话历史上下文
- 当前可用工具列表
- 每个工具的名称、描述、参数 schema、示例
- 模型最终选择的工具
- 模型生成的参数
- Ground Truth（正确工具与正确参数）
- 错误标签
- 可解释特征向量

如果未来要做多步 agent，还可以继续加入：
- 中间工具返回结果
- 下一步决策
- 状态转移信息

但你当前阶段建议先做**单步工具调用**，也就是研究“用户请求 → 工具选择 + 参数生成”这一层。

### 2. 任务库设计
任务库建议按四类构造：

#### （1）清晰直接型
例如：
- “查询订单 ID 为 12345 的配送状态”
- “把 2025-04-01 的航班 MU123 状态查一下”

这类样本主要用于构造**正确调用基线**。

#### （2）模糊歧义型
例如：
- “帮我看看那个订单怎么样了”
- “查一下上周那个航班”

这类样本用于研究：模型在信息不完整时，是不是会错误选择工具。

#### （3）参数不完整型
例如：
- “更新一下订单状态”
- “帮我改一下客户资料”

这类样本主要用来研究：模型是否会**参数幻构**，或者是否会正确进行澄清。

#### （4）误导干扰型
例如：
- “去数据库查一下这个品牌 CEO，顺便看看网上有没有新闻”
- “帮我查订单，网页上也许能找到”

这类样本用于制造工具间竞争，研究模型为什么会选错。

### 3. 工具库设计
建议先定义 10 个工具，分成三组：

#### （1）相似工具组
比如：
- `search_web`
- `search_database`
- `query_internal_docs`

这组主要用来研究**工具混淆问题**。

#### （2）复杂参数工具组
比如：
- `book_flight(date, flight_no, origin, destination)`
- `update_order(order_id, status, note)`
- `create_ticket(user_id, category, priority, description)`

这组主要用来研究**参数幻构问题**。

#### （3）低歧义对照组
比如：
- `get_weather(city)`
- `translate_text(text, target_lang)`

这组主要用于对照：在结构简单时，错误率是否显著下降。

### 4. 数据规模建议
初期目标：
- 至少 2000 条样本
- 其中错误样本占比不低于 40%
- 至少两个模型参与生成
- 最好覆盖两种语言以上

### 5. 标注体系
标注至少分两层：

#### 第一层：结果标签
- `0 = 正确`
- `1 = 工具选错`
- `2 = 参数幻构`

如果后续更细化，还可以扩展成：
- 缺参数但应澄清未澄清
- 参数类型错
- 参数值伪造
- 调用了高风险工具

#### 第二层：人工可读特征标签
这部分是整个研究的核心，因为后续因果图就是建立在这些特征之上。

## 三、特征体系设计

建议将特征划分为三类：**指令侧、工具侧、匹配侧**。

### 1. 指令侧特征
研究“用户怎么说”会不会诱发错误。

建议特征包括：
- 动词是否明确
- 参数是否显式出现
- 是否存在指代词（“那个”“这个”“上次那个”）
- 是否存在歧义实体
- 是否包含多个潜在任务
- 是否夹带无关信息
- 是否是低资源语言/非英语表达
- 指令长度
- 关键词密度

### 2. 工具侧特征
研究“工具是怎么定义的”会不会导致混淆。

建议特征包括：
- 工具名称长度
- 工具名与功能是否一致
- 工具描述与相邻工具的语义相似度
- 参数 schema 的层级深度
- 参数是否存在类型约束
- 是否提供示例值
- 是否区分高权限/低权限
- 描述是否冗长或模糊

### 3. 匹配侧特征
研究“请求和工具之间的关系”是否有问题。

建议特征包括：
- 用户意图与正确工具描述的语义相似度
- 用户意图与错误工具描述的语义相似度
- 正确工具与错误工具的混淆分数
- 参数缺失率
- 参数名与用户文本对齐度
- 是否存在误导性高相似工具

### 4. 特征实现方式
你不需要全部人工打标，建议三种方式结合：

- **规则提取**：比如参数是否缺失、指令长度、是否有日期格式等
- **Embedding 计算**：比如用户意图与工具描述的相似度
- **LLM 辅助判别**：比如“该请求是否模糊”“是否存在歧义实体”

最终目标是形成一个 **20~30 维的特征向量**。

## 四、模型结构设计

你的模型建议采用“双路径结构”：

### 路径一：因果结构学习模块
目标：找出哪些特征在因果意义上驱动错误。

#### 基础版
先用经典因果发现算法做基线：
- PC
- NOTEARS
- DirectLiNGAM（可选）

节点包括：
- 20~30 个人工可读特征
- 错误标签节点（正确 / 工具错 / 参数幻构）

你需要输出：
- DAG 图
- 到错误节点的直接父节点
- 关键因果路径

#### 增强版
引入 LLM 语义表征：
- 对用户请求做 embedding
- 对工具描述做 embedding
- 加一个特征对齐模块，把 embedding 压到和人工特征同一空间
- 再将对齐后的表示输入因果学习模块

这样你和纯统计方法的区别就出来了：
你不是只看表格特征，而是让模型的语义理解参与因果发现。

### 路径二：表征边界分析模块
目标：研究模型内部“正确 / 错误调用”的边界。

建议操作如下：
- 取模型某一层 hidden state 或最终 tool-call logits 前的表示
- 对样本做降维（PCA / UMAP）
- 观察正确调用与错误调用是否形成可分区域
- 特别关注边界附近样本

边界研究的意义在于：
- 你可以找到“最容易翻转”的样本
- 这类样本最适合做防御训练

### 联合优化思路
最终你可以做一个联合框架：
- 分类头预测错误类型
- 因果头学习 DAG
- 表征边界模块识别危险边界样本
- 用分类损失 + 图稀疏约束 + 对齐损失联合优化

## 五、实验指标设计

### 1. 基础效果指标
- 工具选择正确率
- 参数正确率
- 工具错误率
- 参数幻构率

### 2. 轨迹一致性指标
即使是单步工具调用，也可以加入：
- 是否在缺少关键信息时选择澄清而不是乱填
- 是否在存在多个候选工具时选出 Ground Truth 工具

### 3. 因果解释指标
- 图稀疏度
- 关键边数量
- 关键因果路径长度
- 直接父节点稳定性（多次训练是否一致）

### 4. 反事实验证指标
对某个被识别为直接因的特征进行干预后：
- 错误率是否显著下降
- 参数幻构率是否显著下降
- 工具混淆是否减少

### 5. 边界分析指标
- 边界样本密度
- 正误样本类间距离
- 错误样本到边界的平均距离

### 6. 防御效果指标
- 防御前后工具错误率变化
- 防御前后参数幻构率变化
- 防御前后总体任务成功率
- 是否引入新的误杀（本来能正确调用，结果被拦截）

## 六、消融实验与鲁棒性测试

你至少需要准备以下消融：

### 1. 去掉特征对齐模块
验证 embedding 对齐是否真的提升因果图质量。

### 2. 去掉人工特征，仅用 embedding
验证“可解释特征”是不是必须的。

### 3. 去掉工具侧特征
验证工具定义本身是否真的是错误来源。

### 4. 去掉匹配侧特征
验证请求—工具对齐关系是否重要。

### 5. 不使用边界样本训练，只随机采样
验证边界样本是不是更有效。

### 6. 多模型鲁棒性
至少在两个模型上复现，验证方法是否不是模型特化。

### 7. 多语言鲁棒性
在中文与英文上对比，研究边界与因果图是否漂移。

## 七、创新点与理论贡献

你这篇工作的创新点可以明确写成三条：

### 创新点 1：首个面向工具调用错误的因果可解释性数据集
重点不是“有工具数据集”，而是“有错误标签 + 人工可读特征 + 因果研究友好结构”。

### 创新点 2：融合语义对齐的因果结构学习框架
你不是单纯跑 PC / NOTEARS，而是让高维语义表示和可解释特征联合参与因果发现。

### 创新点 3：基于因果关键因子的防御系统
你不是只解释问题，还把因果图转成了一个可落地的防御模块。

理论贡献在于：
- 将“工具调用错误”从经验现象上升到机制解释；
- 证明错误并非随机，而是由一组可识别、可干预的因子驱动；
- 连接了因果可解释性与 agent 安全。

## 八、风险边界与伦理合规

### 1. 因果不等于绝对真理
你必须把结果表述为“在当前数据与假设下成立的因果结构”，而不是哲学意义上的最终因果真相。

### 2. 数据集偏差
如果样本全是你自己构造的，可能无法覆盖真实世界的复杂调用场景。因此需要：
- 明确标注数据来源
- 发布构造规则
- 留出扩展接口

### 3. 防御不能损害正常可用性
一个“什么都拦住”的系统不是好防御。你必须同时报告：
- 错误率下降
- 误伤率是否上升

### 4. 避免转向攻击工具
你的工作应聚焦“解释 + 校正 + 防御”，不要把重点放在“如何更强地诱导模型调用错误工具”。
